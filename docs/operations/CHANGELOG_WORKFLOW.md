# Changelog Workflow

- Group changes by user-facing areas such as clicker, wallet, polls, docs, and ops.
- Prefer short entries that explain outcome instead of implementation details.
- Link any follow-up work when a release ships with a known limitation.
- Draft notes before deployment so the release summary is ready once checks pass.
- Mention contract names when an entry changes on-chain behavior.
- Reference incident/hotfix IDs when release notes are tied to outage recovery work.
- Include the pull request link for entries that alter support or recovery instructions.

Companion index: [Operations docs](README.md).

### Process Reminder
- Link each release note entry to at least one validating commit hash before publishing.
